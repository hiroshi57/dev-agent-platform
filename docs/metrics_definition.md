# 開発生産性 計測指標 定義書 (draft v0.1)

> ステータス: **仕様書ドラフト（承認待ち）** / 対象: C1 dev-agent-platform / 作成: 2026-07-09

## 0. 目的

Claude Code等の開発エージェント運用を「計測可能」にし、案件別のAI寄与率・削減工数を数値化して、
営業資料（「AI開発体制により従来比◯%短納期」）に**実測で**転用できる状態を作る。

## 1. 測る指標（What we measure）

| 指標 | 定義 | 収集元 |
|------|------|-------|
| AI支援PR比率 | AI寄与判定されたPR / 全PR | Gitメタ＋トレーラー＋申告タグ |
| PRリードタイム | 作成→マージの時間 | GitHub API |
| レビュー往復回数 | PRのレビューコメント/再pushの回数 | GitHub API |
| リバート率 | revert された PR 比率 | Git履歴 |
| 削減工数(四半期) | AI支援あり/なしのリードタイム差×件数 | 上記の集計 |

- AI寄与判定 = コミットトレーラー（Co-Authored-By 等）＋開発者の申告タグの併用。
- 収集は GitHub API（案件リポジトリに **read 権限のみ**）。**コード本文は取得しない**。

## 2. 測らない / 単独評価を禁止する指標（What we do NOT）

- **行数(LOC)ベースの生産性**は指標にしない（誤誘導しやすい）。
  もし表示する場合も、必ずリードタイム・リバート率とセットで表示し、単独評価を禁止する旨を明記。
- 個人ランキング（萎縮・ゲーミングを招くため案件/チーム単位に留める）。

## 3. 誤用リスクと緩和

| 誤用リスク | 緩和策 |
|-----------|-------|
| AI比率だけで生産性を語る | リードタイム・リバート率と常にセット表示 |
| 数値のゲーミング（トレーラー乱用） | リバート率・レビュー往復で品質面を併記 |
| 営業資料での過大主張 | 全数値に出典（集計クエリID）を脚注自動付与、実測のみ生成 |

## 4. 受け入れ基準

- ダミーリポジトリ（履歴付き）で AI支援あり/なしのリードタイム比較が表示されること。
- エージェント使用不可リポジトリで pre-commit 警告が発火すること。
- レポートの全数値に出典（集計クエリID）がひも付くこと。
